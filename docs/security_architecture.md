# معمارية الأمن والحماية (Security Architecture Document)
### منصة LegalOS — نظام إدارة المكاتب القانونية للمستأجرين المتعددين

---

## 1. أمن البيانات وعزلها (Data Security & Isolation)

البيانات القانونية للموكلين هي أصول بالغة الحساسية، لذا يطبق النظام قواعد صارمة لحمايتها:

### 1.1 سياسة عزل الأسطر (Row-Level Security - RLS)
* يتم تطبيق RLS مباشرة على طبقة قاعدة البيانات كحاجز حماية أخير لمنع كشف أي بيانات ناتجة عن خطأ برمجي في التطبيق.
* لا يتم الاتصال بقاعدة البيانات بصلاحيات الـ Superuser (`legalos_admin`) للتطبيقات اليومية، بل يتم استخدام حساب التطبيق المقيد [legalos_app](file:///c:/Users/z2k1s/OneDrive/Pictures/naj1/database/schema.sql#L253-L264).
* يُمنع استخدام استعلامات الحذف الحقيقي (`HARD DELETE`) على أي بيانات مرتبطة بالقضايا، الفواتير، أو ملفات الموكلين؛ ويتم استبدالها بـ `Soft Delete` (عمود `deleted_at`) للامتثال القانوني وسهولة استعادة السجلات.

### 1.2 تشفير الأسرار الحساسة (Envelope Encryption for Najiz Keys)
لحماية مفاتيح ربط ناجز مطورين الخاصة بكل مكتب محاماة:
1. يتم تشفير الأسرار (Secrets) باستخدام خوارزمية التشفير المتناظر **AES-256-GCM**.
2. لا يُستخدم مفتاح تشفير واحد للنظام ككل؛ بدلاً من ذلك، يُستخدم مفتاح تشفير رئيسي (Master Key) يولد مفاتيح تشفير فرعية لكل مستأجر على حدة (Envelope Encryption).
3. تُخزن المفاتيح الرئيسية في نظام خارجي آمن لإدارة المفاتيح (مثل AWS KMS أو HashiCorp Vault) ولا تُكتب أبداً في كود المشروع أو قواعد البيانات المشتركة.

## 2. أمن واجهات الـ API وحماية خوادم الويب (Web & API Security)

### 2.1 تكوين العناوين الأمنية (Helmet & CSP)
يتم تهيئة مكتبة `Helmet` في خادم NestJS لحقن عناوين الحماية الافتراضية للمتصفح:
* `Content-Security-Policy`: تفعيل سياسات صارمة لتحديد مصادر جلب الصور والبرمجيات لمنع هجمات Cross-Site Scripting (XSS).
* `X-Frame-Options: DENY`: منع تضمين منصة LegalOS داخل إطارات (Frames) في مواقع أخرى لمنع هجمات Clickjacking.

### 2.2 التحكم بمعدل الطلبات (Rate Limiting & Throttle)
نستخدم **Redis** لتتبع معدل الطلبات وحجب الهجمات العشوائية:
* **واجهات تسجيل الدخول واستعادة كلمة المرور:** حد أقصى 5 طلبات في الدقيقة لكل عنوان IP أو بريد إلكتروني.
* **واجهات الاستعلام العادية (API Get):** حد أقصى 100 طلب في الدقيقة لكل مستأجر.
* **واجهات التعديل والكتابة (API Post/Patch):** حد أقصى 30 طلب في الدقيقة لكل مستأجر.

## 3. حماية رموز الجلسات (JWT Lifecycle & Session Management)

* **Access Token:** رمز مشفر (JWT) يحتوي على `tenantId`, `userId`, `role`. مدة الصلاحية **15 دقيقة فقط**، ويُرسل في عناوين الطلبات كـ `Bearer Token`.
* **Refresh Token:** يُستخدم لتوليد رموز جديدة دون الحاجة لإعادة تسجيل الدخول. مدة الصلاحية **7 أيام**، ويُخزن في الكوكيز الآمنة الخاصة بالمتصفح (`HttpOnly`, `Secure`, `SameSite=Strict`) لمنع سرقته عبر برمجيات الخبيثة في المتصفح.
